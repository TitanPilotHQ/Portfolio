# Hiring OS — 02. Anticipated Hiring

## Phase B — future (conditional, no date)

> "Initial hires are expected only after successful design-partner
> validation. Early priorities include engineering, customer success,
> product design, and sales. No compensation or equity policy has been
> finalized."
*(Source: `INTAKE-REQUEST.md` §8.2, Emad's direct answer, 2026-07-12, quoted
verbatim.)*

Per Global Constraint 3, this describes a planned future state, framed
conditionally — never as a present-tense claim. Today's actual state (no
hiring at all, Phase A) is documented in `01-current-state.md` and is not
restated here.

## The four anticipated role categories

- Engineering
- Customer success
- Product design
- Sales

Intake §8.2 lists these four as "early priorities" without ranking them,
sequencing them, or indicating which would be hired first. This file
preserves that flat treatment rather than inventing an order — the same
approach Module 12 takes with its six partnership categories
(`12-partnership-os/01-partnership-categories.md`), listed at a general
level with no implied priority among them. Nothing beyond the category name
itself is stated for any of the four here.

## The trigger: "successful design-partner validation"

Intake §8.2 does not define "successful design-partner validation"
precisely — no threshold, no metric, no specific milestone is stated. This
file does not manufacture one, the same way Module 12's
`02-engagement-rules.md` declines to define "formal engagement" beyond the
plain-language boundary the phrase implies.

Module 09 (`09-customer-success-os/03-success-and-production-readiness.md`)
documents the closest existing framing of what "validation" plausibly
relates to within this program: its discussion of "production readiness
review" (Stage 7) as a checkpoint for assessing whether shadow-mode
evidence gathered with a design partner is sufficiently complete to
continue, expand, or eventually support a promotion decision. That file is
explicit that this connection is its own inference, not a criterion Emad
stated for Stage 7 specifically. This file does not go further than Module
09 already did — it does not claim that Module 09's Stage 7 IS,
definitionally, the "successful design-partner validation" referenced in
intake §8.2. No source states that these two phrases refer to the same
event; they are conceptually adjacent (both concern judging whether
design-partner engagement has reached a sufficiency threshold), and this
file notes that adjacency without merging them into a single fact.

## The employment/contracting dependency

A formal hire as an employee typically requires an employer entity capable
of formally employing someone. Module 10
(`10-operations-os/03-legal-entity-and-structure.md`) and Module 07
establish that no legal entity is incorporated yet — that fact is cited
here, not restated. As a structural consequence of that fact alone (not
from any additional source): the employee form of a formal hire cannot
exist until an entity capable of employing one does. This mirrors exactly
how Module 12's `02-engagement-rules.md` handles the same dependency for
signed partnership agreements.

An informal contractor arrangement might not require incorporation the same
way an employee relationship would — that distinction exists in principle.
This file notes the distinction only; it does not invent a plan to use
contractors, since none is sourced. Intake §8.2 says "hires," not
"employees" or "contractors" specifically, and this file does not narrow
that word beyond what is stated.

## What this file deliberately does not invent

- A compensation range, for any role category.
- An equity percentage or equity band, for any role category.
- A title or level structure (e.g., "senior," "junior," "lead," "head of").
- A specific hiring date, quarter, or year.
- A headcount number, for any role category or in total.
- An org chart or reporting structure.
- A benefits framework.
- A remote/in-office or hybrid work policy.
- Job description content for any of the four role categories beyond the
  bare category name itself — no responsibilities, no required experience,
  no seniority, no department structure.
